# 代码结构说明

这份说明对应当前已经整理过的后端目录，目标是让代码结构清晰、提交范围明确、测试文件不进入 git。

## 根目录建议保留

```text
road-crack-detection/
├─ docs/                  文档
├─ scripts/               启动和辅助脚本
├─ server/                后端主工程
├─ .gitignore             忽略规则
├─ README.md              项目说明
└─ 第三章_需求分析_完整版.md
```

## 当前后端主结构

```text
server/
├─ pom.xml
├─ README.md
├─ crack-common/          通用响应体、异常、状态码
├─ crack-api/             枚举、请求 DTO、响应 DTO
├─ crack-dao/             实体、Mapper、MyBatis-Plus 配置
├─ crack-service/         业务服务、算法占位、JWT、业务模型
├─ crack-websocket/       WebSocket/STOMP 推送实现
├─ crack-bootstrap/       启动类、控制器、异常处理、过滤器、配置、SQL 初始化
└─ crack-report/          预留模块，当前仅保留 Maven 模块占位
```

## 各模块职责

- `crack-common`
  - 放通用返回结构：`ApiResponse`、`PageResponse`
  - 放业务异常和结果码

- `crack-api`
  - 放接口层共用的数据结构
  - 例如登录、检测任务、工单、维修报告的 request/response

- `crack-dao`
  - 放数据库实体和 Mapper
  - 负责和 MySQL 表结构对接

- `crack-service`
  - 放核心业务逻辑
  - 包含认证、用户、检测任务、工单、维修报告
  - 包含算法服务占位实现 `MockAlgorithmClient`

- `crack-websocket`
  - 放实时推送相关实现
  - 当前负责检测进度、检测结果、工单更新、告警推送

- `crack-bootstrap`
  - 实际启动模块
  - 放 Controller、过滤器、OpenAPI、CORS、异步线程池、数据库初始化 SQL

- `crack-report`
  - 这是预留模块
  - 当前没有实际业务代码，先保留模块名即可，后续做报表导出再往里补

## 当前应该提交到 git 的内容

- `server/` 下各模块源码
- `server/pom.xml`
- `scripts/start-backend-local.ps1`
- `docs/` 下项目文档
- `.gitignore`

## 当前不应该提交到 git 的内容

- `server/**/target/`：Maven 构建产物
- `*.jar`、`*.class`：打包结果和编译结果
- `temp/`：同学压缩包、临时 SQL、联调脚本、本地测试文件
- `tmp/`：临时文件目录
- `application-local.yml`：本地数据库、邮箱、JWT 等敏感配置

## 本地联调时你会看到但不用提交的内容

- `temp/bankend1-package/`
  - 后端 1 同学原始压缩包内容，只作为参考

- `temp/step1_mysql_setup.sql`
- `temp/step1_detection_smoke.ps1`
- `temp/step2_workorder_report_smoke.ps1`
  - 这些都是本地联调辅助文件，不进入版本库

- `server/*/target/`
  - 编译和打包时自动生成

## 建议你后面维持的整理原则

- 新的业务代码优先放进 `server/` 多模块结构，不要再往 `temp/` 里堆正式代码
- 本地测试数据、截图、SQL 草稿、联调脚本统一放 `temp/`
- 真正需要团队共享的脚本再放到 `scripts/`
- `crack-bootstrap` 只放启动和接口入口，不把大段业务逻辑写进 Controller
- `crack-report` 先保留空模块，避免后续再改父工程结构
