# 数据库设计报告

最后更新：2026-07-07

---

## 一、数据库概述

### 1.1 设计目标

本数据库设计面向“全流程道路裂缝智能检测系统”，以需求文档为依据，目标是在**尽量不冗余**的前提下，覆盖以下核心业务：

- 用户登录、角色区分、部门归属
- 道路与路段基础信息管理
- 监控采集、人工上传、众包上报
- 检测任务与检测结果存储
- 病害明细、工单生成、工单流转
- 维修报告与维修反馈
- 路面健康档案与分析报告
- Agent 问答记录
- 异常预警
- 操作日志与系统配置

### 1.2 设计原则

- 满足需求文档中明确提出的功能，不做过度设计
- 支持第一阶段快速开发，也保留后期扩展空间
- 文件类数据只存路径或 URL，不直接存大对象
- 优先保证主业务链路：
  检测任务 -> 检测结果 -> 工单 -> 维修报告 -> 健康档案
- 权限模型采用**精简版**
  在 `sys_user` 中直接存 `role_code`，暂不拆 `sys_role` 和 `sys_user_role`

### 1.3 技术选型

- 数据库：MySQL 8.0
- 字符集：`utf8mb4`
- 存储引擎：InnoDB
- 主键：`bigint` 自增
- 时间字段：`datetime`
- 逻辑删除：`is_deleted`

### 1.4 命名规范

- 系统管理：`sys_*`
- 道路基础：`road_*`
- 检测业务：`detection_*`
- 工单维修：`work_*`
- 智能分析：`agent_*`、`analysis_*`
- 预警统计：`alert_*`

## 二、总体设计结论

### 2.1 需求复核后的结论

根据需求文档复核，下面这些设计结论比较稳妥：

- `sys_department` 建议保留
  因为需求里明确有部门管理、部门数据隔离、部门专属工作台、工单按部门流转
- `sys_role` 和 `sys_user_role` 可以先不建
  因为课程项目阶段可用 `sys_user.role_code` 代替
- `maintenance_report_file` 可暂时合并到 `maintenance_report`
- `road_health_event` 可暂不单独建
- `stat_daily_summary` 可暂不单独建

### 2.2 最终推荐保留的表

本报告最终推荐保留 **20 张表**，它们刚好覆盖需求文档中的主要功能，又不会像完整产品化方案那样过重：

### 2.3 第一阶段建议优先落地的表

如果你们要先做出一个能跑通的系统，建议先建这 11 张：

1. `sys_user`
2. `sys_department`
3. `road_info`
4. `road_segment`
5. `detection_task`
6. `detection_media`
7. `detection_result`
8. `detection_result_item`
9. `work_order`
10. `work_order_flow`

这 11 张已经能支撑：

- 登录后的人员区分
- 部门派单
- 检测记录
- 病害详情
- 工单流转
- 维修反馈

## 七、为什么这样设计不算冗余

这版设计已经做过一次压缩，主要做了下面这些简化：

- 去掉 `sys_role`
- 去掉 `sys_user_role`
- 去掉 `maintenance_report_file`
- 去掉 `road_health_event`
- 去掉 `stat_daily_summary`

保留下来的表基本都能在需求文档中找到直接对应的功能场景，因此不算“为了设计而设计”。

尤其是这几类表虽然看起来多，但实际上都各有明确用途：

- `sys_department`
  因为需求明确有部门派单、部门隔离、部门专属工作台
- `crowd_submission`
  因为需求明确有众包采集和审核队列
- `road_health_archive`
  因为需求明确有路面健康档案和历史追溯
- `analysis_report`
  因为需求明确有自动生成分析报告
- `alert_rule`、`alert_record`
  因为需求明确有异常预警

## 九、最终推荐结论

如果目标是“刚好涵盖需求文档里的主要功能，同时尽量不冗余”，这版 20 张表是目前最平衡的方案。

其中最关键的取舍是：

- 保留 `sys_department`
- 不拆 `sys_role` 和 `sys_user_role`
- 在 `sys_user` 中直接使用 `role_code`
- 保留健康档案、报告、预警相关表
- 删除明显可以延后或合并的表

这套方案既能解释得通，也方便后续真正落到 Spring Boot + MyBatis-Plus 上。
