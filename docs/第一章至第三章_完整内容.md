# 全流程道路裂缝智能检测系统 — 项目文档

---

## 1 引言

介绍项目的背景、意义和内容，概述了项目的目标和范围。

### 1.1 项目背景

介绍项目的发起原因、现状和问题，以及项目的相关领域和应用场景。

随着我国交通基础设施的快速发展，截至 2025 年底，全国公路总里程已超过 545 万公里，其中高速公路超过 18 万公里。道路在长期使用过程中，受车辆荷载、温度变化、水分侵蚀等因素影响，不可避免地会产生裂缝、坑槽等病害。若不能及时发现和处理，裂缝会进一步发展，导致路面结构破坏，大幅增加修复成本，甚至引发交通安全事故。

当前，国内道路裂缝检测主要依赖人工巡检方式。检测人员徒步或驾车沿路目视检查，手动记录裂缝的位置、类型和严重程度。该方式存在以下突出问题：

（1）**效率低下**：人工巡检速度通常为 5-10 km/h，检测效率低，难以覆盖大规模路网。一条 100 公里的高速公路，人工巡检需要数天时间。

（2）**安全性差**：检测人员在车流密集环境中作业，面临较高交通安全风险，尤其是高速公路，事故隐患突出。

（3）**标准不统一**：检测结果高度依赖个人经验，不同人员对同一裂缝的判定可能得出不同结论，缺乏客观量化标准。

（4）**数据管理困难**：人工记录的数据多为纸质记录或简单电子表格，难以系统化存储、查询和追溯，无法进行历史对比分析和趋势研判。

近年来，无人机技术和人工智能技术取得了突破性进展。无人机搭载高清相机可快速获取道路影像数据，单日巡检里程可达 50-100 公里，效率是人工的 10 倍以上。与此同时，以深度学习为代表的计算机视觉技术，在目标检测、语义分割等领域已达到较高精度，能够自动识别图像中的裂缝目标并计算量化指标。

本项目将无人机视频采集、深度学习裂缝识别、前后端分离开发等技术有机结合，构建一套从数据采集、实时分析到报告生成的全流程闭环系统，为道路养护部门提供智能化检测解决方案。

### 1.2 项目意义

介绍项目的价值和贡献，以及项目的创新点和优势。

本项目的实施具有以下重要价值：

**（1）大幅提升检测效率**
系统支持无人机视频流实时接入与自动化分析，检测速度可达人工巡检的 10 倍以上。对于一条 100 公里的高速公路，传统人工巡检需 3-5 天，使用本系统可在半天内完成数据采集和分析，显著缩短检测周期。

**（2）保障人员作业安全**
以无人机替代人工上路巡检，从源头上消除检测人员在交通环境中作业的安全风险，实现"机器换人、科技兴安"，具有显著的社会效益。

**（3）实现检测标准客观量化**
深度学习模型以统一标准对每一帧图像进行裂缝识别，避免人工检测的主观差异。系统自动输出裂缝的长度、宽度、面积、密度等量化指标，使检测结果可比较、可复现、可追溯。

**（4）科学辅助养护决策**
系统根据裂缝严重程度自动分级，严重病害自动触发 AI Agent 生成详细检测报告，内容包括裂缝分布统计、重点路段标记、成因分析和养护建议，为道路养护部门的资源分配和维修计划提供科学依据，推动养护模式从"被动维修"向"主动预防"转变。

**（5）促进数据资产积累**
所有检测数据系统化存储至数据库，支持按道路、时间等维度进行历史查询和趋势分析，帮助养护部门掌握道路病害发展规律，指导养护资金和资源的科学配置。

### 1.3 项目内容

介绍项目的主要功能和特点，以及项目的预期成果和效果。

本项目开发一套全流程道路裂缝智能检测系统，核心功能包括：

（1）**无人机视频流接入与实时分析**：支持 RTMP/RTSP 等实时视频流协议接入，自动抽帧进行裂缝检测，检测结果实时推送至前端仪表盘。

（2）**自动化裂缝识别与量化评估**：基于深度学习目标检测和语义分割模型，自动识别图像中的裂缝区域，计算裂缝长度、宽度、面积、密度等量化指标。

（3）**历史数据管理与趋势分析**：系统化存储检测任务记录、原始图像和分析结果，支持按道路、时间等维度查询、对比分析和趋势可视化。

（4）**多级告警与 AI Agent 报告生成**：根据裂缝严重程度（轻微/中等/严重）自动分级告警，严重病害触发 AI Agent 自动生成详细检测报告和养护方案。

（5）**前后端分离架构**：前端提供可视化交互界面，后端提供 RESTful API 和 WebSocket 实时通信服务，支持独立部署和横向扩展。

项目预期成果包括：

- 一套完整的道路裂缝智能检测软件系统（含前端、后端、数据处理模块）
- 一个训练好的高精度裂缝检测与分割模型（ONNX 格式，支持 GPU 推理）
- 一份系统技术文档和一份实地部署验证报告

### 2.1 深度学习目标检测技术

#### 2.1.1 技术定义与发展历史

目标检测是计算机视觉领域的核心任务之一，其目标是在图像中定位并分类感兴趣的目标物体。传统目标检测方法（如 Haar 特征 + AdaBoost、HOG + SVM）依赖手工设计的特征，在复杂场景下泛化能力有限。

2012 年，AlexNet 在 ImageNet 图像分类竞赛中取得突破性成果，开启了深度学习在计算机视觉领域的广泛应用。目标检测技术随之进入深度学习时代，经历了以下发展阶段：

| 阶段 | 代表模型 | 核心特点 |
|------|---------|---------|
| 两阶段检测器（2014-） | R-CNN、Fast R-CNN、Faster R-CNN | 先提取候选区域，再分类回归，精度高但速度慢 |
| 单阶段检测器（2016-） | YOLO、SSD、RetinaNet | 直接回归目标位置和类别，速度快、端到端 |
| Anchor-Free（2019-） | CenterNet、FCOS | 无需预设锚点框，简化检测流程 |
| Transformer 时代（2021-） | DETR、RT-DETR | 基于注意力机制，去除 NMS 后处理 |

本项目选用 **YOLOv8** 作为裂缝检测的基础模型。YOLOv8 是 Ultralytics 公司于 2023 年发布的目标检测框架，在 COCO 数据集上实现了精度与速度的良好平衡，适合部署在实时推理场景中。

#### 2.1.2 核心原理

YOLOv8 的核心思想是将目标检测视为一个端到端的回归问题，其网络结构由三部分组成：

- **骨干网络（Backbone）**：采用 CSPDarknet 结构，通过跨阶段部分连接（Cross Stage Partial）提取多尺度特征，在降低计算量的同时保持特征表达能力。
- **颈部网络（Neck）**：使用 FPN+PAN 结构，融合不同尺度的特征图，使模型能够同时检测大目标和小目标（裂缝属于小目标，需要高分辨率特征图的支持）。
- **检测头（Head）**：采用解耦头（Decoupled Head），分别输出目标的分类概率和边界框坐标，采用 Task-Specific 学习策略提升检测精度。

#### 2.1.3 在道路裂缝检测中的应用

将 YOLOv8 应用于道路裂缝检测，需进行以下适配：

- **数据集构建**：采集不同光照、不同路面类型（沥青/水泥）下的裂缝图像，标注裂缝的边界框位置和类别（横向裂缝/纵向裂缝/网状裂缝/坑槽）。
- **模型微调**：使用预训练权重（在 COCO 或 ImageNet 上预训练），在自建的裂缝数据集上进行迁移学习，以较少的数据量即可获得较好的检测效果。
- **小目标优化**：裂缝在图像中通常占比较小，可通过 Mosaic 数据增强、多尺度训练等策略提升模型对小目标的检测能力。

**优势**：推理速度快（在 RTX 3060 上可达 100+ FPS），满足实时检测需求；模型轻量，支持 ONNX 导出和 TensorRT 加速。

**挑战**：对密集微小裂缝的检测仍有漏检可能；在不同光照和路面纹理条件下泛化性需要持续优化。

### 2.2 语义分割技术

#### 2.2.1 技术定义与发展历史

语义分割是对图像进行像素级分类的技术，即为图像中的每个像素分配一个类别标签。与目标检测只输出边界框不同，语义分割能够精确勾勒出目标区域的轮廓，是裂缝量化评估（计算宽度、面积等指标）的关键技术。

语义分割技术的发展历程如下：

| 阶段 | 代表模型 | 核心特点 |
|------|---------|---------|
| 全卷积时代（2015-） | FCN、SegNet | 用卷积层替换全连接层，实现像素级预测 |
| Encoder-Decoder（2015-） | U-Net、DeepLab 系列 | 编码器提取特征，解码器恢复分辨率，跳跃连接保留细节 |
| 注意力机制（2018-） | DANet、CCNet | 引入自注意力机制捕捉长距离依赖 |
| Transformer（2021-） | SETR、SegFormer | 将 Transformer 结构用于分割任务 |

本项目选用 **U-Net** 作为裂缝语义分割的基础模型。U-Net 最初于 2015 年提出用于医学图像分割，因其优秀的编码器-解码器结构和跳跃连接设计，被广泛应用于各类分割任务。

#### 2.2.2 核心原理

U-Net 的网络结构因其对称的 U 形而得名，由以下两部分构成：

- **编码器（下采样路径）**：由多个卷积层和池化层组成，逐步缩小特征图尺寸、增加通道数，提取图像的高层语义特征。编码器的每一层输出都会通过跳跃连接传递到解码器的对应层。
- **解码器（上采样路径）**：由反卷积或上采样层组成，逐步恢复特征图尺寸至输入分辨率，输出每个像素的类别概率。
- **跳跃连接（Skip Connection）**：将编码器每一层的特征图与解码器对应层的特征图拼接，使解码器能够同时利用高层语义信息和底层细节信息，确保分割结果的边缘精度。

#### 2.2.3 在道路裂缝量化评估中的应用

U-Net 在裂缝分割中的具体应用流程：

- **模型输入**：原始道路图像（或 YOLOv8 检测出的裂缝区域裁剪图）。
- **模型输出**：与输入图像同尺寸的分割掩码图，每个像素被分类为"裂缝"或"背景"。
- **量化指标计算**：基于分割掩码，通过图像处理算法计算裂缝的像素面积、骨架长度（细化算法）、最大宽度（距离变换）等指标。结合标定参数（像素-毫米换算），将像素尺寸换算为物理尺寸。

**典型模型**：本项目基于 U-Net 改进的 CrackU-Net 结构，在原 U-Net 基础上引入残差连接和注意力门控机制（Attention Gate），提升了裂缝边缘的检测精度。

**优势**：能够精确提取裂缝区域的像素级轮廓，为宽度测量、面积计算提供基础；在小样本数据集上仍能取得较好效果。

**挑战**：推理速度相对较慢（与 YOLO 相比）；对噪声敏感，路面上的污渍、水渍、阴影等可能造成误分割；需要足够的标注数据（像素级标注成本高）。

**流程图说明：**

1. **前置条件**：用户已登录系统，拥有检测员或管理员角色权限。
2. **步骤一（配置视频源）**：用户在视频流管理页面输入 RTMP/RTSP 地址或选择本地视频文件，设置抽帧频率（默认 2 帧/秒）、检测区域（ROI）和告警阈值。
3. **步骤二（启动检测任务）**：用户点击"开始检测"，系统创建检测任务实例，进入实时检测循环。
4. **步骤三（循环检测）**：系统按配置的抽帧频率从视频流中读取帧图像，送入裂缝检测模型（YOLOv8+U-Net）进行推理，得到裂缝位置、分割掩码和量化指标。
5. **步骤四（结果处理）**：检测结果通过 WebSocket 实时推送至前端，视频画面上叠加裂缝标注框和掩码，仪表盘数据动态更新。同时检测结果写入 PostgreSQL 数据库。
6. **步骤五（用户交互）**：用户可随时暂停/停止检测，查看当前检测详情。检测过程中若检测到严重裂缝，系统自动弹窗告警。
7. **后置条件**：检测任务停止后，所有检测数据已持久化存储，可供历史查询和报告生成使用。

```
 │  ├─ 定时触发：每日/每周自动汇总          │
 │  └─ 手动触发：用户点击"生成报告"         │
 └──────────────────┬───────────────────────┘
                    ↓
 ┌──────────────────────────────────────────┐
 │        收集检测数据                      │
 │  ├─ 从数据库读取检测任务所有记录         │
 │  ├─ 统计：总检测帧数、裂缝总数、         │
 │  │        各级别裂缝数量、最大宽度等     │
 │  └─ 获取代表性检测图像（含标注）         │
 └──────────────────┬───────────────────────┘
                    ↓
 ┌──────────────────────────────────────────┐
 │        裂缝严重度分级                    │
 │                                         │
 │  ┌────────────┐                         │
 │  │ 轻微裂缝   │ ← 最大宽度 < 0.2mm     │
 │  │ 仅记录归档  │                         │
 │  └────────────┘                         │
 │  ┌────────────┐                         │
 │  │ 中等裂缝   │ ← 0.2mm ≤ 最大宽度      │
 │  │ 生成简要报告│    ≤ 0.5mm             │
 │  └────────────┘                         │
 │  ┌────────────┐                         │
 │  │ 严重裂缝   │ ← 最大宽度 > 0.5mm     │
 │  │ 生成详细报告│                         │
 │  └────────────┘                         │
 └──────────────────┬───────────────────────┘
                    ↓
 ┌──────────────────────────────────────────┐
 │    AI Agent 报告生成                     │
 │  （严重级别时自动执行）                   │
 │  ├─ 分析裂缝分布规律                    │
 │  ├─ 标记重点养护路段                    │
 │  ├─ 生成养护建议方案                    │
 │  └─ 生成报告文本内容                    │
 └──────────────────┬───────────────────────┘
                    ↓
 ┌──────────────────────────────────────────┐
 │        报告预览与审核                    │
 │  ├─ 用户在线预览报告内容                 │
 │  ├─ 支持人工编辑和补充说明               │
 │  └─ 审核通过→标记为"已审核"             │
 └──────────────────┬───────────────────────┘
                    ↓
 ┌──────────────────────────────────────────┐
 │        报告导出                          │
 │  ├─ 导出为 PDF 格式（供打印/存档）       │
 │  └─ 导出为 Word 格式（供编辑/修改）      │
 └──────────────────────────────────────────┘
```

**流程图说明：**

1. **前置条件**：至少有一个已完成的检测任务，且检测数据已存储至数据库。
2. **步骤一（触发条件判断）**：报告生成由三种方式触发：① 自动触发——实时检测中检测到严重裂缝时自动发起；② 定时触发——系统按预设周期（每日/每周）自动汇总生成；③ 手动触发——用户在历史数据页面点击"生成报告"。
3. **步骤二（数据收集）**：系统从数据库中读取指定检测任务的全部检测记录，进行统计汇总，包括总检测帧数、裂缝总数、各级别裂缝数量、最大裂缝宽度、平均裂缝宽度等。
4. **步骤三（严重度分级）**：根据裂缝的最大宽度进行三级分类：轻微（<0.2mm）、中等（0.2-0.5mm）、严重（>0.5mm）。不同级别采用不同的后续处理策略。
5. **步骤四（AI Agent 报告生成）**：对于严重级别的检测结果，AI Agent 自动分析裂缝分布规律，标记重点养护路段，并生成科学的养护建议方案。报告内容涵盖检测概况、裂缝分布统计、严重路段标记和养护建议。
6. **步骤五（人工审核）**：生成的报告支持在线预览，审核员可对报告内容进行人工编辑和补充说明，审核通过后标记为"已审核"。
7. **步骤六（报告导出）**：支持导出为 PDF（供打印存档）和 Word（供进一步编辑修改）两种格式。
8. **后置条件**：报告已生成并存储至数据库，用户可随时查看、导出或删除历史报告。

### 3.3 功能需求说明

对每个功能模块进行详细说明，给出了功能的目标、条件、约束、异常处理等信息，并给出了功能的用例图、活动图等。

#### 3.3.1 功能模块一：实时检测模块

**模块描述：**

实时检测模块是本系统的核心业务模块，负责视频流的接入管理、实时帧分析、裂缝检测推理和检测结果的实时展示。该模块贯穿从视频源接入到检测结果呈现的完整链路，是系统功能闭环中的关键环节。

**角色描述：**

| 角色 | 角色说明 | 可操作功能 |
|:----:|---------|-----------|
| 检测员 | 日常执行检测任务的操作人员，负责视频流配置、检测启停和结果查看 | 配置视频源、启动/暂停/停止检测、查看实时数据、查看检测历史 |
| 审核员 | 对检测结果进行核实和审阅的人员 | 查看实时检测详情、查看检测结果标注、审核告警信息 |
| 管理员 | 系统最高权限用户，管理检测参数和系统配置 | 设置检测参数默认值、管理告警阈值、查看所有检测任务 |

**用例描述：**

| 用例名称 | 启动实时检测 |
|---------|-------------|
| **用例概述** | 用户配置视频流源地址和检测参数，启动实时裂缝检测任务，系统开始读取视频帧并进行裂缝检测，检测结果实时推送至前端展示 |
| **前置条件** | ① 用户已登录系统，拥有检测员或管理员角色；② 系统后端服务运行正常；③ 视频流源可用（RTMP/RTSP 服务器已推流或视频文件已上传） |
| **后置条件** | ① 检测任务状态变为"运行中"；② 前端仪表盘开始实时更新检测数据；③ 检测结果持续写入数据库 |

| 用例名称 | 查看实时检测结果 |
|---------|-----------------|
| **用例概述** | 用户在检测过程中查看实时展示的检测结果，包括视频画面上的裂缝标注叠加层和仪表盘上的量化指标数据 |
| **前置条件** | ① 用户已登录系统；② 当前有正在运行的检测任务 |
| **后置条件** | 用户可直观了解当前道路的裂缝检测状况，包括裂缝位置、数量、严重程度等 |

| 用例名称 | 处理检测告警 |
|---------|-------------|
| **用例概述** | 当检测到严重裂缝时，系统自动触发告警，前端弹窗提示并发出声音警报，提醒用户关注 |
| **前置条件** | ① 检测任务正在运行；② 当前帧检测到的裂缝宽度超过告警阈值 |
| **后置条件** | ① 前端显示告警提示框；② 该条检测记录被标记为"已告警"；③ 若设置了自动报告，系统开始收集数据准备生成报告 |

#### 3.3.2 功能模块二：报告管理模块

**模块描述：**

报告管理模块负责基于检测数据自动生成检测报告，支持多级告警触发、AI Agent 智能分析和多格式导出。该模块将分散的检测数据整合为结构化的分析报告，为道路养护决策提供科学依据。

**角色描述：**

| 管理员 | 系统管理员，可配置报告模板和告警规则 | 配置报告模板、设置告警阈值、管理报告的增删改查 |

### 3.4 非功能需求说明

对每个非功能需求进行详细说明，给出了需求的目标、度量、验证方法等信息，并给出了需求的质量属性模型等。

#### 3.4.1 性能需求

**需求描述：**

性能需求规定了系统在响应时间、吞吐量、资源利用率等方面应达到的指标，是保证系统在实际使用场景中能够稳定、高效运行的关键质量属性。

**质量属性模型：**

```
性能质量属性
├── 时间特性
│   ├── 响应时间：用户操作到系统反馈的时间间隔
│   ├── 处理延迟：视频帧输入到检测结果输出的时间
│   └── 吞吐量：单位时间内处理的视频帧数量
├── 资源利用率
│   ├── CPU 占用率
│   ├── GPU 占用率
│   ├── 内存占用率
│   └── 网络带宽占用
└── 容量
    ├── 支持的最大并发用户数
    ├── 支持的最大视频流路数
    └── 数据库存储容量上限
```

**详细指标要求：**

| 需求项 | 指标要求 | 度量方法 | 验证方法 |
|-------|---------|---------|---------|
| 实时检测帧率 | ≥ 15 FPS | 统计连续 5 分钟的平均检测帧数 | 使用性能监控脚本记录帧率日志 |
| 单帧检测延迟 | ≤ 200ms（P95） | 从帧图像送入模型到结果返回的时间 | 统计 1000 帧检测延迟的 P95 分位值 |
| 视频流接入延迟 | ≤ 3 秒 | 从视频源推流到前端画面显示的时间 | 使用时间戳测量对比 |
| 历史查询响应时间 | ≤ 2 秒 | 10 万条记录内按条件查询的返回时间 | 使用测试脚本模拟并发查询 |
| 报告生成时间 | ≤ 30 秒 | 单次检测任务全量报告从触发到生成完成的时间 | 测试不同数据量下的生成耗时 |
| 系统并发用户 | ≥ 10 人 | 同时在线操作系统的用户数 | 使用 JMeter 等压测工具模拟并发 |
| 系统可用性 | ≥ 99.5% | 月累计正常运行时间占比 | 监控系统运行时间，统计故障时长 |
| 7×24 小时连续运行 | 无内存泄漏、无服务崩溃 | 内存占用率不持续增长 | 长时间运行监控测试 |

**异常处理：**

- 当 GPU 显存不足时，系统自动降低抽帧频率并给出提示
- 当检测延迟超过阈值时，系统自动熔断并记录告警日志
- 当数据库连接超时时，查询请求返回友好错误信息，避免前端无响应

#### 3.4.2 精度与安全性需求

**需求描述：**

精度需求规定了裂缝检测模型在识别准确率、测量误差等方面的技术指标，是衡量系统核心检测能力的关键依据。安全性需求规定了系统在用户认证、数据保护、通信安全等方面的保障要求。

**（一）精度需求指标**

| 需求项 | 指标要求 | 度量方法 | 验证方法 |
|-------|---------|---------|---------|
| 裂缝检测精确率（Precision） | ≥ 90% | TP / (TP + FP) | 使用人工标注测试集进行评测 |
| 裂缝检测召回率（Recall） | ≥ 85% | TP / (TP + FN) | 使用人工标注测试集进行评测 |
| 裂缝检测 F1 分数 | ≥ 0.87 | 2 × (P × R) / (P + R) | 综合精确率和召回率计算 |
| 裂缝宽度测量误差 | ≤ ±0.1mm | 测量值与实际值之差（标定后） | 使用已知宽度的标准裂缝样本验证 |
| 误报率 | ≤ 5% | 非裂缝区域被标记为裂缝的比例 | 在无裂缝路面图像上测试 |
| 漏检率 | ≤ 10% | 未被检测出的裂缝占总裂缝的比例 | 使用人工标注测试集统计 |

**（二）安全性需求指标**

| 需求项 | 指标要求 | 验证方法 |
|-------|---------|---------|
| 用户身份认证 | 密码加密存储（bcrypt 算法），登录失败 5 次后暂时锁定账号 15 分钟 | 安全测试用例验证 |
| 权限控制 | 基于 RBAC 的权限管理，不同角色只能访问授权功能和数据 | 遍历测试各角色的 API 权限 |
| 通信安全 | 前后端通信使用 HTTPS 协议（TLS 1.2+），WebSocket 使用 WSS 协议 | 使用 Wireshark 检查通信是否明文 |
| API 鉴权 | 所有业务接口使用 JWT Token 鉴权，Token 有效期 24 小时 | 发送无 Token 请求验证是否被拒绝 |
| SQL 注入防护 | 使用 ORM 参数化查询，过滤用户输入的特殊字符 | 使用 SQLMap 工具进行注入测试 |
| XSS 防护 | 前端对用户输入数据进行转义处理，禁止直接渲染 HTML | 使用 XSS 测试 payload 验证 |
| 数据备份 | 数据库每日自动备份，保留最近 30 天备份文件 | 检查备份任务日志和备份文件完整性 |
| 日志审计 | 关键操作（登录、检测启动/停止、报告审核等）记录操作日志，包括操作人、时间、IP 和操作内容 | 触发关键操作后查询日志表验证 |

**异常处理：**

- 认证服务异常时，已登录用户的会话不受影响，但新登录请求返回错误提示
- 超出登录失败限制后，账号自动锁定并发送通知邮件（可选功能）
- 数据传输过程中若检测到证书异常，前端给出明确的安全警告并中断连接
